# Library Export Design — Single-Doc and Bulk ZIP

**Date:** 2026-05-28
**Status:** Approved (brainstorm)
**Builds on:** v0.7.2

## Goal

Give the Library a first-class export, symmetric with how tasks already export from the Admin panel:

1. **Single-doc** — from the document editor toolbar, download the open doc as a `.md` file with round-trippable YAML front-matter. Fully client-side.
2. **Bulk** — from the Admin panel, download a project (or all projects) as a ZIP of `.md` files. Server endpoint streams via `archiver`.

Both paths emit the same `.md` shape so that the single-file output and the ZIP entries are interchangeable.

## Current state (context)

- Tasks already have export ([public/js/renderAdminPanel.js](public/js/renderAdminPanel.js)): per-project + all-projects buttons that produce `moomora.tasks` JSON envelopes, filenames generated by `exportFilename` in [public/js/importExport.js](public/js/importExport.js).
- Library has **no** export of any kind. Import of `.md` files exists and preserves the original filename in the `source_filename` column.
- Document shape in [server/schema.sql](server/schema.sql): `title`, `body`, `document_type` ('runbook' | 'note'), `project_id`, `tags text[]`, `source_filename`, `archived_at`, timestamps.
- Library routes ([server/libraryRoutes.js](server/libraryRoutes.js)) already resolve project slug → id via `projectsRepository.resolveProject`; the new export route reuses that path.
- Demo server ([scripts/demo-server.js](scripts/demo-server.js)) registers the same routes against in-memory repos — a new route shows up in demo automatically; the in-memory library repo needs the one new method.

## Non-goals (v1)

- No export of archived documents. Archived rows are skipped from bulk export. Single-doc export is only surfaced in the editor for active docs (archived docs render read-only in the archive view and won't get an export button in v1).
- No re-import of the new `.md` files. Round-trip is a follow-up — the front-matter is designed to support it but the import path is not changed in this feature.
- No client-side ZIP (kept out for memory/bundle reasons — see Approach below).
- No JSON envelope export for the library (skipped in favour of the `.md`-first format).
- No `Content-Length` header on the streaming response (`archiver` streams; we emit the headers we control and let the connection close).

## File shape (the contract)

Each exported document is a `.md` file beginning with a YAML front-matter block, a blank line, then the body verbatim, with a guaranteed trailing newline:

```markdown
---
title: Postgres restore
type: runbook
project: homelab
tags:
  - postgres
  - dr
created_at: 2026-04-12T10:33:21Z
updated_at: 2026-05-20T08:11:09Z
---

# Postgres restore

…body…
```

- All six fields always present.
- `tags` always emitted as a block sequence; an empty tag list renders as `tags: []`.
- Strings that contain `:`, `"`, `\`, or a leading `-` are emitted as double-quoted YAML with `\` and `"` escaped.
- `created_at`/`updated_at` are emitted as the ISO-8601 strings the DB returns, unmodified.
- Body is written byte-for-byte from the row, with a trailing `\n` appended if absent.

### Filename rules

Used identically by the single-doc download and by each ZIP entry:

1. If `source_filename` is set and non-empty, basename-sanitize it: strip any path separators (`/`, `\`), strip leading dots, collapse to the final segment. If the result still ends in `.md` (case-insensitive), use it; otherwise append `.md`.
2. Else, slugify `title`: lowercase, replace any run of non-`[a-z0-9]` with `-`, trim leading/trailing `-`. Append `.md`.
3. If the sanitized/slugified result is empty, fall back to `untitled.md`.

### ZIP structure

- Per-project export → flat: each entry is `<filename>.md`.
- All-projects export → grouped: `<project-slug>/<filename>.md`.
- Entry collision within the ZIP (same filename inside the same folder) → append `-2`, `-3`, … before `.md`, in iteration order.

### Download filenames

Mirroring the existing `exportFilename` convention in [public/js/importExport.js](public/js/importExport.js):

- Per-project ZIP: `moomora-console-library-<project-slug>-YYYY-MM-DD.zip`
- All-projects ZIP: `moomora-console-library-all-YYYY-MM-DD.zip`
- Single-doc: the computed filename from the rules above (no date prefix — it's a single document, not a backup).

## Backend

### New module: `server/libraryExport.js`

Pure functions, no I/O — all the logic that gets unit-tested lives here.

- `formatFrontMatter(doc, projectSlug)` → string. YAML-escapes title; emits tags as a block sequence (or `tags: []`); emits timestamps as-is.
- `renderDocumentMarkdown(doc, projectSlug)` → full `.md` string (front-matter + blank line + body + trailing newline).
- `documentFilename(doc)` → safe filename per the rules above.
- `dedupeFilenames(entries)` → mutates the list of `{ filename, path }` entries to apply `-2`, `-3` suffixes for collisions within the same `path` prefix.

### Repository (`server/libraryRepository.js`)

One new method:

- `listActiveDocumentsForExport({ projectId })` → returns all non-archived docs joined with their project slug (`select d.*, p.slug as project_slug from markdown_documents d join projects p on p.id = d.project_id where d.archived_at is null [and d.project_id = $1] order by p.slug, d.title`). `projectId` undefined → all projects. Returns normalized rows with `projectSlug` set.

### Route (`server/libraryRoutes.js`)

One new endpoint:

```
GET /api/library/export?project=<slug>|all
```

Behaviour:

1. If `project` is missing, empty, or `all`: scope = all projects. Otherwise resolve via `projectsRepository.resolveProject`; 400 `{message: "project is invalid"}` if unknown (same shape as existing routes).
2. Fetch active docs via `listActiveDocumentsForExport`.
3. Set headers:
   - `Content-Type: application/zip`
   - `Content-Disposition: attachment; filename="<computed>.zip"` (filename from `exportFilename`-style helper, but library-flavoured — added to a small server-side helper module rather than importing the browser helper).
4. Open an `archiver` instance (`archiver('zip', { zlib: { level: 6 } })`), pipe to `reply.raw`, append each doc as `renderDocumentMarkdown(doc, doc.projectSlug)` at the entry path (flat for per-project, `<project-slug>/` prefix for all), finalize.
5. Empty result set → finalize an empty (but valid) ZIP — the user always gets a download.
6. On `archiver` `error` mid-stream: log via `app.log.error`, call `reply.raw.destroy()` to terminate cleanly. Headers are already flushed; we can't return JSON at that point.

### Dependency

Add `"archiver": "^7.0.1"` to `dependencies` in [package.json](package.json); `npm install` updates the lockfile.

### Demo server

In [scripts/demo-server.js](scripts/demo-server.js), add `listActiveDocumentsForExport({ projectId })` to the in-memory library repo. Implementation: filter `documents` by `!archivedAt` and optional `projectId`, then attach `projectSlug` by lookup against the in-memory projects array. No other change — the route is shared and `archiver` runs the same way against either backend.

## Frontend

### New module: `public/js/libraryExport.js`

Pure browser helpers — separate implementation from the server module (Node vs browser), tested against the same fixtures so the two stay byte-identical.

- `buildExportedMarkdown(doc, projectSlug)` — same output bytes as `renderDocumentMarkdown` on the server.
- `documentFilename(doc)` — same rules as the server helper.
- `triggerDownload(filename, blob, { documentRef = document })` — creates an anchor, sets `download` + object URL, clicks, revokes the URL.

### Single-doc export — editor toolbar

In [public/js/renderLibrary.js](public/js/renderLibrary.js), the document editor toolbar gets an `[x] export` bracket-button (next to the existing actions) with `data-action="export-document"` and `data-document-id="<id>"`. Rendered only for active docs (the archive view stays read-only).

In [public/js/main.js](public/js/main.js), wire `export-document`:

1. Look up the doc in `state.documents` by id.
2. Look up the project slug from `state.projects` by `doc.projectId`.
3. Call `buildExportedMarkdown(doc, projectSlug)`, wrap in a `Blob`, call `triggerDownload(documentFilename(doc), blob)`.

No fetch, no server change.

### Bulk export — Admin panel

In [public/js/renderAdminPanel.js](public/js/renderAdminPanel.js), add a new `Library` section above the existing Tasks `Backup` section:

```
Library
  <project-name> · <active-doc-count> documents · Generated at download time
  [x] export <project-name>   [X] export all
```

- Buttons: `data-action="export-library-project"` and `data-action="export-library-all"`.
- Active doc count is computed from `state.documents` (filtered to non-archived, optionally scoped to the current project). No extra API call.
- The Tasks `Backup` and `Restore / Import` sections are untouched.

In [public/js/main.js](public/js/main.js), wire both actions to:

```js
window.location.href = `/api/library/export?project=${encodeURIComponent(scope)}`;
```

…where `scope` is the current project slug (per-project button) or `'all'`. The browser handles the download via the response's `Content-Disposition` header — no fetch/blob juggling needed.

## Error handling

**Single-doc (client-side):**

- Doc not found in `state` (defensive — the button is rendered from state): log to console, no-op. Not user-visible.
- Project slug not resolvable from `state.projects`: fall back to the literal `unknown` in front-matter rather than failing the download. The file still opens cleanly in any editor; if the user re-imports later and `unknown` isn't a real project, the import flow will reject or prompt — that's the import path's problem to solve, not export's.

**Bulk export endpoint:**

- `project` present but unresolvable → `400 {message: "project is invalid"}`.
- Empty active doc set → `200` with a valid empty ZIP (consistent UX).
- `archiver` `error` mid-stream → log + `reply.raw.destroy()`. Truncated download client-side is acceptable for a homelab tool; server logs are the primary signal.
- DB error before streaming starts → standard Fastify `500` (headers not yet sent, normal JSON error response).

**Filename safety:**

- `documentFilename` always strips path separators and leading dots — a malicious `source_filename` like `../../etc/passwd` becomes `etc-passwd.md`. ZIP slip is impossible because we control the entry path server-side.
- Empty/whitespace-only result → `untitled.md`.

## Data flow summary

```text
Single-doc:
  click [x] export in editor
    → buildExportedMarkdown(doc, slug)
    → triggerDownload(documentFilename(doc), blob)
  (no network)

Bulk:
  click [x] export <project> or [X] export all in Admin
    → window.location.href = '/api/library/export?project=<slug|all>'
    → GET /api/library/export
        → resolve project (or all)
        → listActiveDocumentsForExport
        → archiver streams entries → reply.raw
    → browser saves <Content-Disposition filename>.zip
```

## Build order

1. **Shared shape — backend pure module + tests**: `server/libraryExport.js` (`formatFrontMatter`, `renderDocumentMarkdown`, `documentFilename`, `dedupeFilenames`) with unit tests covering escape/edge cases.
2. **Repository method + demo mirror**: `listActiveDocumentsForExport` in [server/libraryRepository.js](server/libraryRepository.js) and the in-memory version in [scripts/demo-server.js](scripts/demo-server.js).

4. **Single-doc frontend**: `public/js/libraryExport.js`, editor toolbar button in [public/js/renderLibrary.js](public/js/renderLibrary.js), `export-document` handler in [public/js/main.js](public/js/main.js), frontend tests.
5. **Bulk frontend**: `Library` section in [public/js/renderAdminPanel.js](public/js/renderAdminPanel.js), two handlers in [public/js/main.js](public/js/main.js), render tests.

## Testing

### Backend

`tests/backend/libraryExport.test.js` — unit tests for the pure module:

- `formatFrontMatter`:
  - Title with `:` is double-quoted.
  - Title containing `"` and `\` is escaped.
  - Empty `tags` renders as `tags: []`.
  - Multi-tag list renders as a block sequence.
  - ISO timestamps emitted verbatim.
- `renderDocumentMarkdown`:
  - Output ends with exactly one trailing newline.
  - Body containing a literal `---` line is preserved unmodified (front-matter delimiters do not collide).
  - Runbook vs note types both round-trip.
- `documentFilename`:
  - Uses `source_filename` when present, normalised.
  - Sanitizes `../foo.md` → `foo.md`.
  - Slugifies title when `source_filename` is null.
  - Empty/whitespace title → `untitled.md`.
- `dedupeFilenames`:
  - Three `foo.md` entries → `foo.md`, `foo-2.md`, `foo-3.md`, in order.
  - Collisions are scoped per ZIP folder (same name under different project-slug prefixes does not collide).

`tests/backend/libraryExportRoute.test.js` — route test against the Fastify app + in-memory repos used by other backend tests:

- `GET /api/library/export?project=all` → `200`, `Content-Type: application/zip`, `Content-Disposition` filename matches the all-projects pattern. Decode the ZIP inline (small reader covering the EOCD + local file headers — `archiver`'s output is well-formed STORE/DEFLATE so this stays ~50 lines). Assert entry paths are `<project-slug>/<filename>.md` and each body contains the expected front-matter.
- `GET /api/library/export?project=homelab` → entries are flat, only homelab docs present.
- `GET /api/library/export?project=does-not-exist` → `400 {message: "project is invalid"}`.
- Empty library → `200`, ZIP decodes to zero entries.

If the inline ZIP reader grows past ~50 lines, the fallback is to inject a fake `archiver` and assert on `append` call arguments instead. The pure module is already byte-tested, so the route test only needs to prove "the right inputs reach archiver in the right shape".

### Frontend

`tests/frontend/libraryExport.test.js`:

- `buildExportedMarkdown` — same fixtures as the backend `renderDocumentMarkdown` tests, asserting identical byte output. This pins the two implementations together.
- `documentFilename` — same fixtures as backend, identical output.
- `triggerDownload` — using a mock `document` ref (pattern from existing `openTaskImportFilePicker` tests), assert anchor created, `download` attribute set, `click()` called, object URL revoked.

`tests/frontend/renderAdminPanel.test.js` (or the existing render tests, depending on layout) — assert the new `Library` section renders with the project name, doc count, and both buttons with the expected `data-action` attributes.

`tests/frontend/renderLibrary.test.js` (or wherever editor-toolbar rendering is tested) — assert the `[x] export` button renders on the editor toolbar for active docs and does not render in the archive view.

### Manual smoke (PR description)

1. `npm run demo` → Library → edit a doc → `[x] export` → file downloads, opens in any markdown viewer with intact front-matter.
2. Admin → `Library` → `[x] export <project>` → ZIP downloads, contents are flat `.md` files.
3. Admin → `Library` → `[X] export all` → ZIP contains `<project-slug>/...` subfolders.
4. Empty library project → ZIP downloads and is a valid empty archive.

## Risks / notes

- New dependency (`archiver`) — small, stable, well-maintained, justified for streaming correctness.
- Two implementations of the `.md` serializer (Node module + browser module) intentionally kept separate to avoid bundling Node code into the browser; the shared-fixture test suite is what keeps them in lockstep. If they drift, the fixture test fails.
- Round-trip on import is not delivered here. The front-matter is designed to support it but the import code is unchanged. A follow-up can add a `.md`-with-front-matter import mode without revisiting the export format.
- Demo server change is one method addition — small, mirrors the production repo signature, keeps the demo as the primary manual smoke surface.
- Archived docs are intentionally excluded from bulk export to match the tasks export's "live data" intent. Backup of archived content is a separate problem (database-level backup or a future archive-aware export mode).
